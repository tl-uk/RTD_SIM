ARCHITECTURE SUMMARY (confirmed correct — do not change)
================================================================================

Data flow:
  bdi_planner.actions_for()
    → For STRUCTURED_MODES (rail/tram/ferry):
        TripChainBuilder.build() → [access_leg, trunk_leg, egress_leg]
        each leg routed via env.compute_route_with_segments()
        result: 3-segment TripChain with real boarding/alighting points
    → For ROUTABLE_MODES (car/ev/bus/walk/bike):
        env.compute_route_with_segments() directly
        result: 1-segment TripChain
    → TripChain stored in params['trip_chain']
    → cognitive_abm._maybe_plan() propagates to agent.state.trip_chain
    → simulation_loop.py copies all multimodal fields to agent_states dict
    → visualization.py renders per-leg PathLayers

Graph separation (CRITICAL — never merge):
  drive graph   — OSMnx network_type='drive'
  walk graph    — OSMnx network_type='walk'
  bike graph    — OSMnx network_type='bike'
  rail graph    — OpenRailMap Overpass query
  tram graph    — OSM railway=tram (sidings pruned, retain_all=False)
  transit graph — GTFSGraph
  ferry graph   — ferry_network.fetch_maritime_graphs()  ← NOT rail_network shim
  shipping_lanes — OpenSeaMap TSS (visualisation only)

All coordinates: (lon, lat). Never (lat, lon).

Key attribute: SpatialEnvironment.router (Router instance, set at line 94)
  → Used by TripChainBuilder._snap_stop() via getattr(env, 'router', None)
  → Router.snap_to_transit_stop() is a public method (added session 9)

─────────────────────────────────────────────────────────────────────────────
ISSUE 1 — EV MODE BIAS (~56 EV out of ~170 agents = 33%)
─────────────────────────────────────────────────────────────────────────────

Current: EV 56/170 = 33% (latest log: simulation_20260421_234648.log)
Target:  ~20-25%

Root cause: When no GTFS is loaded, tram/rail routing falls back to the OSM
graph which may fail catchment checks or return empty routes. EV always
succeeds (drive graph is always available). This creates EV as the universal
fallback winner.

With TripChainBuilder now wired, rail/tram will succeed more often because:
  - Stop snapping is precise (GTFS/NaPTAN platform coords)
  - Access/egress use correct modes (walk if <1.2km, car if >1.2km)
  - Catchment guard correctly filters to TMU-only stops

Expected: next run with TripChainBuilder active should reduce EV by 5-10%.
If EV is still >30%: check CPG mode selection for budget_student persona
(EV=0.0 in personas.yaml but CPG still adds it via 'bus' fallback logic).

─────────────────────────────────────────────────────────────────────────────
ISSUE 2 — FERRY GRAPH RENDERING
─────────────────────────────────────────────────────────────────────────────

Current: Ferry graph loads as hardcoded spine (Overpass fails with 406)
Expected after fix: Overpass great-circle arc routes

After the Content-Type fix, Overpass should succeed. If Overpass is still
failing, check:
  1. Network connectivity from the server running the simulation
  2. Whether Overpass is rate-limiting (look for 429 in logs)
  3. Cache age: delete ~/.rtd_sim_cache/transport/ to force re-fetch

─────────────────────────────────────────────────────────────────────────────
ISSUE 3 — GTFS TRAM GEOMETRY (no shape data)
─────────────────────────────────────────────────────────────────────────────

When GTFS is loaded from BODS (TransXChange conversion), Edinburgh Tram
has no shape.txt entries. Tram routes show 4.5 pts/stop (straight lines)
vs bus routes at 27.7 pts/stop (real road geometry).

Fix needed in router.py _compute_gtfs_route() or _gtfs_with_segments():
  When shape_coords is empty for a tram segment, fall back to the OSM
  tram graph geometry between the nearest tram-track nodes.

The OSM tram graph is already loaded as graphs['tram'] — the fix just needs
to use it as a shape source when GTFS has no shapes.

─────────────────────────────────────────────────────────────────────────────
ISSUE 4 — TripChainBuilder: trunk leg geometry is still a stub
─────────────────────────────────────────────────────────────────────────────

TripChainBuilder._build_rail_like() returns a trunk leg with path=
[boarding_stop_coord, alighting_stop_coord] — two points only.

When bdi_planner routes this leg via compute_route_with_segments(), the
router's _intermodal_with_segments() takes over and produces real track
geometry because it calls _nearest_rail_node() on the coords. So the trunk
leg actually gets real geometry through the router.

HOWEVER: the access and egress legs also go through compute_route_with_segments
with the original mode (walk/car). This should work correctly since those
leg origins/dests are now real stop coordinates, not arbitrary agent positions.

Verify in the next log:
  - "TripChain local_train Xkm 3 legs" (3 segments)
  - access leg: "walk route: N pts" or "access→drive proxy: N pts"
  - trunk leg: "local_train intermodal-segments Xkm (N pts, 3 legs)"
  - egress leg: similar to access

If trunk leg routing fails (empty route from stop-to-stop): check that
_intermodal_with_segments can snap both stop coords to rail graph nodes.
The NaPTAN RLY stop coordinates should be close to OpenRailMap nodes.
