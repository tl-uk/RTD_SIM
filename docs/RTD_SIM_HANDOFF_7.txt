PROJECT OVERVIEW
----------------
RTD_SIM is a UK-wide agent-based transport decarbonisation simulator (Edinburgh
development/test focus). Stack: Python, OSMnx, NetworkX, GTFS/BODS, Streamlit/
pydeck, BDI agents. Models modal shift across road / rail / tram / subway / bus / ferry /
air modes for decarbonisation policy research.

Goal: Every agent route must follow real physical infrastructure — roads on the
road graph, rail on OpenRailMap tracks, trams on tram tracks, subway on subway lines, ferries on ferry
routes. Multi-modal agents must show colour-coded segments per transport mode.
OSM must render ferry/shipping routes. Routing must work for the entire UK
including the Highlands and Islands.

CRITICAL — ROUTING VISUALISATION (agent routes do not follow infrastructure):

  PROBLEM 1: No-GTFS rail routes still show straight diagonal segments.
  CAUSE: With walk graph now loaded, access/egress legs should be fixed. But
    the "track topology fragmented" warnings indicate agents still snap to
    disconnected rail sub-graphs (sidings, buffer stops). LCC extraction in
    load_rail_graph() should address this — needs verification after next run.
  STATUS: Fix implemented (LCC extraction in spatial_environment.py). Not yet
    confirmed working in test — needs a fresh run with the new files.

  PROBLEM 2: With-GTFS bus routes show straight-line segments crossing sea.
  CAUSE A: GTFS shape_coords absent for some stop pairs (dead-run trips, BODS
    feed quality issues). Bus fallback is road proxy which correctly routes on
    roads — but the road graph has no path across the Forth, so the proxy
    produces a 2-point fallback line.
  CAUSE B: Some routes genuinely cross the Forth by bus (E1 airport express
    via Forth Road Bridge). The road proxy SHOULD work for these once the
    bbox captures the Ferrytoll P&R stop. bbox is now 0.3° padding — needs
    verification.
  FIX NEEDED: In router.py _compute_gtfs_route, for bus segments where the
    road proxy fails (returns []), try routing via the drive graph including
    both sides of the Forth. The drive graph does include the Forth Road Bridge
    and Queensferry Crossing — verify the road proxy is attempting to use them.

  PROBLEM 3: Multi-modal agent route colour does not change per transport mode.
  CAUSE: The visualiser/visualization.py PathLayer currently colours entire
    route by the agent's final mode choice. Each route segment needs to carry
    mode metadata for per-segment colouring.
  FIX NEEDED: router.py must return a structured route object (or annotated
    coordinate list) that tags each segment with its mode. visualization.py
    must split the PathLayer per mode segment and apply MODE_COLORS_HEX.
  FILES TO CHANGE: simulation/spatial/router.py, visualiser/visualization.py,
    possibly simulation/route_alternative.py.

  PROBLEM 4: Ferry agent (1 generated in default test) does not appear on map.
  CAUSE A: Ferry routes use _compute_gtfs_route → if no GTFS, _transit_fallback
    returns [] (correct — no road proxy for water). Agent has no route → not
    visualised.
  CAUSE B: Even with GTFS, ferry stops may not have shape_coords in the BODS
    feed, and the straight interpolation between stops is then filtered out
    as a "crossing water" invalid route. No ferry routes visible at all.
  FIX NEEDED:
    (a) OSM ferry/shipping routes need to be downloaded and added as a separate
        'ferry' graph layer in GraphManager. OSM has highway=ferry and
        route=ferry relations for all UK ferry crossings.
    (b) Router needs a _compute_ferry_route() method that routes on this layer
        (like _compute_intermodal_route does for rail).
    (c) Visualisation needs a ferry PathLayer with the correct colour (teal).

  PROBLEM 5: OSM display does not show ferry and shipping routes.
  CAUSE: environment_setup.py only loads network_type='drive', 'walk', 'bike'.
    OSMnx has no 'ferry' network_type — ferry routes in OSM are encoded as
    relations with route=ferry and must be fetched via a custom Overpass query.
  FIX NEEDED: Add fetch_ferry_routes() in rail_network.py (alongside
    fetch_rail_graph) using ox.graph_from_xml or overpass query for
    route=ferry relations. Register as graph_manager.graphs['ferry'].
    Add ferry layer to pydeck visualisation.

  PROBLEM 6: GTFS auto-loading from Transitland not working.
  STATUS: User is using static GTFS files. Transitland integration has not been
    implemented. Remains TODO.
  FIX NEEDED: Implement gtfs_loader.py fetch_from_transitland() method using
    the Transitland v2 API (https://www.transit.land/api/v2/). Key parameters:
    bbox, agencies filter (Lothian Buses: o-gcpv-lothianbuses, ScotRail, etc.)
    API key required (free tier available). Feeds are returned as GTFS zip URLs
    which can then be passed to the existing GTFSLoader.
